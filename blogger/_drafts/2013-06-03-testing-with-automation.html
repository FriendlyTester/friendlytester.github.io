I saw a conversion on twitter again recently, that reminded me off something I wanted to write when James and Michael's "Checking vs. Testing" was redefined, but I completely forgot about it, but here goes!

My current job title according to my contract is "Test Automation Specialist", I have requested this be changed, but still waiting... so by definition my job is:

Test: A procedure intended to establish the quality, performance, or reliability of something, especially before it is taken into widespread use.

Automation: The use or introduction of automatic equipment in a manufacturing or other process or facility.

Specialist: a person who concentrates primarily on a particular subject or activity; a person highly skilled in a specific and restricted field:
he is a specialist in psychometric testing.

therefore:

I am a highly skilled person concentrating on using automatic processes to create a procedure that can be used to establish the quality, performance or reliability or our products.

Interesting isn't it, now lets ignore the specialist bit.

We get "An automatic procedure intended to establish the quality, performance, or reliability of something.

So some of you will be sat there now saying that's exactly what my automation does, does it? Are you sure? Is is actually possible to automate a test?

Because I believe you do the testing part either prior to deciding to automate something or whilst you write the code, and then used the automation to assist your testing.

If you are writing the automation once the feature is established you / someone would have already manually tested it, and gathered information regarding this feature, then all you are doing is creating something to validate that this information is still valid.

If you this is a new feature and you are going straight to automation, you will test this as you are creating it, you will be on the pages finding the locators, using the screen in order to create the script, testing the pages.

So therefore I don feel test automation is a true reflection of what it is we are actually creating, based on the testing vs checking post by James Bach and Michael Bolton I have to agree with this post and that what we are actually creating are 'Automated Checks', these checks are checking known information is still accurate.

Now there is another side to this , which I am going to label 'Testing with Automation', and this is using automation to assist your testing. 

I originally went with something I had heard on twitter 'Tools Assisted Testing', but then I thought about this a bit more, and decided that I have never tested without a tool, I always use my brain and 95% of the time pen and paper, so again this doesn't give automation the praise it deserves, also automation is a collection of tools used to achieve something.

So what is 'Testing with Automation' it's exactly what it says using automaton to assist your testing, for instance write scripts to go and gather data from the application which you can use for your test, a good use of this is tables.

I have been using this approach for a while, but the first time I heard it at a conference was from Matt Archer at TestBash, he actually mentioned the same tool I use, a C# package called HTML Agility pack, he used it to visit pages within his application and download the table content, and create some basic HTML pages with the content, which were then saved in a central location allowing him to open them one by one and validate the content.

Some of you will be thinking that you could automate that into a check, and you would be right, but is it always worth the time and effort, as it can be slow dynamically navigating a table as I found out, and ended up creating this Table Control Object, but our application can have very large data sets and it was simply to slow on the larger tables, so was better to create a script to do the tedious work then use the best tool of all to validate the data in the tables, the brain.

Another example is taking a screenshot after every step/page change, and saving this to a central location then manually validate them.
Once you have done this once though and fixed any issues you have created something to check against, so you can then save the new images and open them up against the previous run for comparison, this is obviously easier if you have two screens!!!

So in conclusion I feel we need to distinguish between these types of automation, and educate people on the differences so they get a better understanding of the power of automation and the many different ways it can be used be used in Testing.

In my opinion automation is invaluable, automated checks relieve testers of spending days executing checks they have executed umpteen times, no one enjoys this, but checks are certainly required and automating these means faster feedback and faster confidence levels.

This gives us testers more time to be creative and explorative with our testing, and find ways to use automation to assist this.
